# JobForge Frontend — Phase 3A

The JobForge web app. Consumes the FastAPI backend at
`http://127.0.0.1:8000` via a Vite dev proxy at `/api`.

## Run

```bash
# from /frontend
npm install
npm run dev     # http://localhost:3000

# in another shell, from the repo root:
uv run uvicorn jobforge.api.main:app --reload
```

## Test

```bash
npm test        # 52 tests, ~3s
```

## Production build

```bash
npm run build   # vite build && tsc --noEmit
npm run preview # serve dist/ on :3000
```

## Tech stack

- **TanStack Router** (file-based routing under `src/routes/`)
- **React 19** + **TypeScript** (strict)
- **Tailwind CSS v4** (CSS-first config in `src/styles.css`)
- **shadcn/ui-style primitives** (hand-rolled on Radix; no codegen runtime)
- **TanStack Query** (server state)
- **Zustand** (theme + toast stores)
- **Vitest** + **React Testing Library** (component + route tests)

## Architecture

```
src/
  main.tsx                  bootstrap: QueryClient, RouterProvider, Toaster
  routes/                   file-based routes (one file per page)
    __root.tsx              shell layout
    index.tsx               Dashboard
    jobs.index.tsx          Jobs list
    jobs.$jobId.tsx         Job detail
    applications.index.tsx  Applications list
    applications.$applicationId.tsx  Application detail
    companies.index.tsx     Company lookup
    skill-gaps.index.tsx    Skill gaps + 7d/30d plan
    resume.index.tsx        Resume upload
    settings.index.tsx      Preferences form
    telegram.index.tsx      Bot status + test digest
  components/
    app-layout.tsx          sidebar nav + mobile menu + theme toggle
    job-card.tsx            JobCard (reusable)
    application-timeline.tsx  ApplicationTimeline + StatusBadge
    match-score-badge.tsx   MatchScoreBadge
    skill-gap-card.tsx      SkillGapCard
    company-card.tsx        CompanyCard
    ui/                     shadcn-style primitives
      button, card, badge, dialog, tabs, table, dropdown-menu,
      inputs (Input/Textarea/Label/Select/Checkbox), toaster, state
  lib/
    api.ts                  fetch wrapper + typed response shapes
    cn.ts                   tailwind class merger
    theme.ts                Zustand: dark-mode store, persists to localStorage
    toast.ts                Zustand: in-app toast queue
  routeTree.gen.ts          generated by the TanStack Router Vite plugin
  styles.css                Tailwind v4 entrypoint + theme tokens
```

## Routes ↔ API endpoints

| Route | Backend endpoint(s) |
| --- | --- |
| `/` | `GET /dashboard`, `GET /jobs/top-matches?limit=5`, `GET /applications?limit=5` |
| `/jobs` | `GET /jobs`, `GET /jobs/top-matches`, `POST /jobs/sync`, `POST /applications` |
| `/jobs/$jobId` | `GET /jobs/{id}`, `GET /jobs/{id}/match`, `GET /companies/{name}`, `POST /tailor`, `POST /applications` |
| `/applications` | `GET /applications`, `PATCH /applications/{id}/status` |
| `/applications/$applicationId` | `GET /applications/{id}`, `PATCH /applications/{id}/status` |
| `/companies` | `GET /companies/{name}` |
| `/skill-gaps` | `GET /skills/plan` |
| `/resume` | `POST /profile`, `GET /dashboard` (presence flag) |
| `/settings` | `GET /preferences`, `PUT /preferences` |
| `/telegram` | `GET /telegram/status`, `POST /telegram/test-digest` |

## Backend changes

Only two backend changes were made (both inside `src/jobforge/api/`):

1. **CORS** added to `main.py` for `http://localhost:{3000,5173}` so the
   SPA can call the API in dev. Existing `RequestIdMiddleware` still runs
   inside the CORS layer.
2. **`routes/telegram.py`** added with `GET /telegram/status` and
   `POST /telegram/test-digest`. Wraps existing `build_digest_data` and
   `deliver_daily_digest`. No new business logic.

All 199 backend tests still pass.

## Design system

shadcn/ui-style: each primitive is a small typed React component built on
Radix primitives + Tailwind, sitting in `src/components/ui/`. Tokens are
declared as CSS custom properties on `:root` and `.dark` so a single
`.dark` class flip switches every color in the app.

Reusable domain components (per the spec):

- **MatchScoreBadge** — colors by bucket (high ≥75 / mid ≥50 / low <50)
- **JobCard** — company / title / location / match score / missing skills + View / Save / Apply
- **ApplicationTimeline** — chronological event list, flags unusual transitions
- **SkillGapCard** — skill / frequency / clamped importance bar
- **CompanyCard** — growth + risk + remote policy + apply recommendation

## Accessibility & responsiveness

- Sidebar collapses to a top-bar menu under `md`
- All interactive elements have visible focus rings via the global
  `*:focus-visible` rule in `styles.css`
- Theme toggle and nav have labeled icon buttons
- Forms use semantic `<label>` via the `Label` primitive
- Dialog and dropdown use Radix (full keyboard nav, ARIA wired up)

## Remaining technical debt

1. **TanStack Router, not Start.** The spec called for TanStack Start
   (SSR). The app uses TanStack Router in SPA mode because the
   FastAPI backend already serves the data API and SSR adds infra cost
   without a clear win for a single-user tool. Migrating later is
   mechanical (move `main.tsx` into a Start entry, keep all routes).
2. **Bundle size warning.** Single 503 kB JS chunk. Acceptable for a
   single-user internal tool; if it matters, switch to route-level
   `lazy` imports — TanStack Router supports this out of the box.
3. **Tailored-resume history.** `POST /tailor` returns an artifact_id
   but there is no `GET /artifacts` endpoint, so the Resume page can
   only show the *current* tailoring session, not a history list.
   Adding that to the backend is a Phase 3 polish.
4. **Company lookup is single-record.** No backend endpoint lists all
   known companies, so `/companies` is a name-typed lookup rather than
   a browseable directory.
5. **Match-sort uses an in-process scan.** When the discovery catalog
   gets large, `GET /jobs/top-matches?limit=100` will become slow.
   PRD-blessed; deferred to a future caching/indexing pass.
6. **No file persistence of tailored resumes.** Downloads are generated
   client-side from the dialog response — once closed, they're gone.
   Re-fetch the artifact server-side when adding history.
7. **No auth.** Single sole-user mode per the backend's `sole_user_id`
   config. Multi-user is Phase 3.
8. **Telegram "Last Digest" surfaces fresh data, not the actually-sent
   one.** The backend has no `last_sent_at` field; the page renders
   `build_digest_data()` output as a preview. Add a `digest_runs` table
   if you want true delivery history.
